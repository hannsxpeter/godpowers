# MRR Tracker Architecture

## Options Considered

| Shape | Score | Why rejected | Trap flag |
|-------|-------|--------------|-----------|
| Local web app over an exported JSON file (baseline) | 8.3 | [DECISION] Chosen, because local import satisfies the Privacy NFR and the weekly review scope without network credentials. | [HYPOTHESIS] Hidden cost: account changes after export stay invisible until the next local import. |
| Live Stripe OAuth with a hosted sync worker | 6.9 | [DECISION] Rejected, because a hosted worker would move customer account names outside the local workspace, contrary to the Privacy NFR. | |
| Stripe webhook receiver with a persistent event store | 5.8 | [DECISION] Rejected, because P-SHOULD-01 asks for a weekly summary, so streaming ingestion adds public ingress and replay handling without improving that weekly flow. | [HYPOTHESIS] Hidden cost: a webhook receiver must be publicly reachable and replay-safe, which is more operational surface than a weekly review workflow justifies. |

[DECISION] Scores include trap demotion and use the NFR map below; novelty is
not a scoring axis.

## System Context

[DECISION] MRR Tracker runs as a single local web app that imports Stripe subscription events, computes weekly movement buckets, and renders a founder review dashboard.

## Containers

- [DECISION] The ingestion container reads Stripe subscription events from an exported JSON file during the example flow.
- [DECISION] The movement engine container normalizes subscription deltas into new, expansion, contraction, and churn buckets.
- [DECISION] The dashboard container renders bucket totals and account drill-down tables.

## NFR-to-Architecture Map

| NFR | Architectural choice |
|---|---|
| Accuracy | [DECISION] The movement engine stores each account delta and sums buckets from those deltas so totals can be reconciled within 1 dollar. |
| Privacy | [DECISION] The example keeps imported Stripe data in the local workspace and does not send account names to a hosted service. |

## Tradeoffs

[DECISION] The example uses local JSON import instead of live Stripe OAuth because the Privacy NFR keeps account names local, and instead of webhooks because the weekly review flow does not need streaming ingestion.

## ADR-001 Local Stripe Export

[DECISION] ADR-001 chooses local Stripe JSON export because it beats hosted OAuth on the Privacy NFR and beats webhook ingestion on the weekly review scope; the tradeoff is snapshot staleness between imports.
